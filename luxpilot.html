LuxPilot — Governed LLM Advisory System

A governed AI advisory system that separates deterministic routing from probabilistic generation.

What it is

LuxPilot is a governed LLM-powered advisory system for Luxembourg microenterprises. It helps small business owners translate messy business context into feasible, explainable marketing action plans while respecting real constraints such as capacity, trust, budget, language, and execution burden.

The system is designed as both a commercial AI product and a research-grade evaluation instrument.

Why it matters

Generic LLM advice is fluent, but it is often not governable. It may recommend actions that sound impressive but exceed the user's capacity, ignore feasibility constraints, or change behavior silently when the model or prompt changes.

LuxPilot addresses this by separating the system into three layers: versioned domain knowledge; deterministic, LLM-free governance logic; and model-agnostic LLM execution contracts. The LLM can extract, explain, and generate. It does not choose the strategic route.

Architecture

LuxPilot's target architecture separates the deterministic and probabilistic parts of the system.

Layer 1 contains versioned domain knowledge: variables, campaign families, feasibility gates, constraints, templates, question banks, and language rules.

Layer 2 contains the deterministic routing engine. It evaluates confirmed profile snapshots against versioned rules and gates. It produces traceable, replayable routing results.

Layer 3 contains model-agnostic LLM execution contracts for extraction, inference, boundary interpretation, explanation, generation, and adaptive questioning.

User input → confirmation → immutable snapshot → deterministic routing → LLM generation → validation → audit/replay

Deterministic / Probabilistic Boundary

Everything that enters the routing engine must be frozen, confirmed, and hashed. LLM outputs can only enter the system as candidates awaiting confirmation or drafts awaiting validation.

This design prevents the model from silently replacing the governance logic. It also makes routing decisions auditable and reproducible.

Evaluation and Reproducibility

LuxPilot is designed around evaluation from the start:

  • immutable profile snapshots;
  • rule-base versioning;
  • golden-trace fixtures;
  • same-input-same-output tests;
  • audit logs for routing executions and LLM calls;
  • replay contract: snapshot hash + rule-base version + engine build → output hash;
  • validation events for user confirmations and generated assets.

The goal is not only to produce useful advice, but to make the system's behavior inspectable, testable, and governable.

What changed from the beta

The beta proved the product flow, but it was not reproducible enough for research-grade evaluation. The rebuild moves routing out of frontend code, replaces hard-coded rules with versioned rules-as-data, decomposes a monolithic prompt into contracted LLM functions, introduces profile snapshot immutability, and replaces funnel analytics with audit/replay infrastructure.
